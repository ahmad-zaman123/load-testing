# Backend Contract

This harness depends on a documented contract with `easychef-backend`. If
the backend changes any of these, the harness needs a matching update.

## Authentication

Each load-test user authenticates via a JWT access token (rest_framework_simplejwt).
The token is installed two ways defensively:

- **Cookie**: `my-app-auth=<access_token>` (matches `JWTCookieAuthentication`)
- **Header**: `Authorization: Bearer <access_token>`

The token fixture (`fixtures/tokens.json`) is generated by the backend's
`seed_load_test_users` management command and has this shape:

```json
{
  "count": 500,
  "default_password": "LoadTest!Pass123",
  "tokens": [
    {
      "user_id": "<uuid>",
      "email": "loadtest+0001@example.com",
      "access": "<jwt access token>",
      "refresh": "<jwt refresh token>"
    }
  ]
}
```

## Cookie names

- JWT access token cookie: `my-app-auth`
- JWT refresh token cookie: `my-refresh-token`

## LOAD_TEST_MODE expectations

When `LOAD_TEST_MODE=true` on the backend:

- 12 external clients are mocked (FCM, LLM via litellm, OpenAI Assistant, Gemini,
  Midjourney, FDC, Barcode, ScraperAPI, Google Sheets, Apple OAuth, Google OAuth,
  Langfuse). Mocks return canned data with simulated latency.
- Mocks are applied at Django startup via `easychef.load_testing.apps.LoadTestingConfig.ready()`.
- The backend hard-fails if `LOAD_TEST_MODE=true` and `ENV=prod` — prod protection.

## Backend shortcuts for load testing

These backend behaviors only activate when `LOAD_TEST_MODE=true` AND `ENV != "prod"`.
Both are gated by 3 layers (env flag + prod-refuse guard + scoped check) so they
can't leak into real user flows.

### OTP shortcut (verify-otp)

`/users/auth/verify-otp/` accepts the fixed value `"0000"` for any email matching
`loadtest+*@example.com` when `LOAD_TEST_MODE=true`. Real emails still go through
real OTP validation regardless of the flag.

The harness's `OnboardingToFirstCookJourney` uses `LOAD_TEST_OTP="0000"` by default.
Override via env var if the backend value ever changes.

### Email backend mock

`send_email` (SendGrid wrapper in `easychef.communications.utils`) is replaced with
a no-op when `LOAD_TEST_MODE=true`. This prevents OTP-issue endpoints
(`/users/auth/resend-otp/`) from returning 500 when SendGrid is unconfigured in dev /
staging. The mock returns a canned `{"status_code": 202, "message_id": "mock-..."}`
shape so callers that inspect the response don't crash.
