# Backend Contract

This harness depends on a documented contract with `easychef-backend`. If
the backend changes any of these, the harness needs a matching update.

## Authentication

Each load-test user authenticates via a JWT access token (rest_framework_simplejwt).
The token is installed two ways defensively:

- **Cookie**: `my-app-auth=<access_token>` (matches `JWTCookieAuthentication`)
- **Header**: `Authorization: Bearer <access_token>`

The token fixture (`fixtures/tokens.json`) is generated by the backend's
`seed_load_test_users` management command and has this shape:

```json
{
  "count": 500,
  "default_password": "LoadTest!Pass123",
  "tokens": [
    {
      "user_id": "<uuid>",
      "email": "loadtest+0001@example.com",
      "access": "<jwt access token>",
      "refresh": "<jwt refresh token>"
    }
  ]
}
```

## Cookie names

- JWT access token cookie: `my-app-auth`

## LOAD_TEST_MODE expectations

When `LOAD_TEST_MODE=true` on the backend:

- 12 external clients are mocked (FCM, LLM via litellm, OpenAI Assistant, Gemini,
  Midjourney, FDC, Barcode, ScraperAPI, Google Sheets, Apple OAuth, Google OAuth,
  Langfuse). Mocks return canned data with simulated latency.
- Mocks are applied at Django startup via `easychef.load_testing.apps.LoadTestingConfig.ready()`.
- The backend hard-fails if `LOAD_TEST_MODE=true` and `ENV=prod` — prod protection.

## Known friction points

### OTP verification in journey tests

The `verify-otp` endpoint requires the OTP value generated by `EmailOTP.generate_or_get_otp()`,
stored in the database with a TOTP secret. The harness has no way to read this from outside
the backend's DB.

**Three resolutions** (we picked option 2 below for the example journey):

1. **DB query from a privileged channel** — harness queries the EmailOTP table directly
   over the staging DB. Couples harness to DB schema; needs DB credentials.
2. **Backend shortcut** — when `LOAD_TEST_MODE=true`, `/users/auth/verify-otp/` accepts a
   known value (e.g. `"0000"`) for any `loadtest+*@example.com` email. ~10-line backend
   patch, gated by env flag and email prefix. The current `OnboardingToFirstCookJourney`
   assumes this shortcut exists.
3. **Skip OTP verification entirely** — only test post-onboarding flows using seeded users
   (see `ReturningUserCookJourney`).

If you run the onboarding journey without resolution 2, the OTP step will 4xx and the
journey aborts on that step. The journey is still useful for measuring `register` and
`resend-otp` throughput.
